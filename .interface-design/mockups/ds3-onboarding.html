Design notes
Design decisions captured in this mockup (v3)
1. No onboarding wizard. Every screen in the "onboarding" is the navigation system at a different scale. Empty states guide the user forward. The interface reveals itself as structure grows — empty scaffolding is never shown. Chrome (scale panel, breadcrumb bar) appears when it has content.

2. Add Data uses natural field labels. Milestone, Source, File — three fields that construction professionals understand immediately. The snapshot triple (when/who/what) informs the architecture, not the UI copy. Every import, first or hundredth, uses the same surface.

3. Type-to-create combobox. Milestones and sources are created inline during import. Type a name that doesn't exist → "Create '100% DD'" appears. No separate creation flows, no modals. The combobox is both search and creation. A2: Milestone ordinal is assigned via AEC pattern matching — the create option shows the detected ordinal ("ordinal 300"). Placeholder examples use standard AEC conventions (25% DD, 50% CD, 100% CD).

4. First import always confirms mapping (A1). Amended from auto-accept. First import from a new source always shows the mapping confirmation screen — framed as "Cadence matched your columns, confirm" rather than "fix these errors." High confidence = stamp checkmarks, fast scan, one click. Builds trust. Subsequent imports from the same source skip this entirely (stored mapping).

5. Context pre-fill. Opening Add Data from a milestone view pre-fills the Milestone field. Opening from a source view pre-fills Source. The form adapts to where you are in the graph.

6. Login rhymes, doesn't match. Same Geist type, same vellum/sheet warmth, centered card composition. Not the universal template — the only screen outside the navigation system. Email/password for MVP.

7. Project list is the root scale. Projects are items of type "project." The project list is the Powers of Ten root — you're looking at projects the way you'd look at rooms in a building. Cards for visual weight, inline create form (no modal) for project creation.

8. Progressive chrome reveal. First-run (no projects): bare vellum + one button. Project list: breadcrumb bar appears. Project view: scale panel + story panel appear. The interface unfolds with the data. Never show empty drawers.

9. "Add Data" shifts from CTA to quiet (A6). On an empty project: primary styling (bg-ink text-sheet) in the breadcrumb bar + central CTA in story panel. Once milestones exist: quiet styling (text-graphite border-rule), blending with chrome. The story panel CTA disappears. Visual emphasis shifts from "add data" to "explore."

10. Import progress, not spinner (A5). Named stages that narrate what the system did: creating milestone, creating source, importing items, detecting changes, checking conflicts. The final summary ("50 items, 0 conflicts") stays visible before navigating. Post-completion animation driven by the API response data — honest and educational.

11. Post-import lands at project dashboard (A3). After import, navigate to the project view — the aerial/dashboard scale. The user sees the executive summary first, then drills down through the scale panel. Powers of Ten from the top down. The project dashboard is the natural starting point for understanding what was just imported.

12. Source type defaults to "schedule" (A4). The source type selector pre-selects "schedule" when creating a new source. The most common first import is a schedule — defaulting correctly removes one decision from the critical path.
