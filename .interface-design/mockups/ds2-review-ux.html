Design notes
Design decisions captured in this mockup (v2)
1. Perspective-dependent conflict display. The main row shows YOUR source's value + a red pip. The pip says "there's a story." Expanding reveals the other side: the conflicting source and its value. Only the conflict item view (pip click → page turn) shows both sources side by side in the comparison grid. This corrects the v1 mockup which showed both sources inline on the main row regardless of perspective.

2. Notes on workflow item view, not inline expansion. The inline expansion (Surface 1) is for quick triage: Resolve or Hold, nothing more. Notes (cairn creation) belong on the workflow item view (Surface 2) where there's room and intent. Notes are orthogonal to status — you can resolve and add a note, hold and add a note, or just add a note.

3. Pips = active workflow items only. No ghost pips. When a workflow item resolves, its pip disappears. Pips are filled (present) or hollow (adjacent), never faded.

4. Cairn = active story without action. A resolved conflict with a pending directive gets a cairn — the decision was made (human voice), the world hasn't caught up yet (directive pending). When the directive is fulfilled, the cairn drops. Full silence. Acknowledged changes go directly to silence — no story left to track.

5. Non-chosen source value in redline-muted. On the conflict item view (Surface 2b), after resolution, the non-chosen source's value is shown in faint redline (redline-muted) instead of neutral trace. A whisper of "this value is noted as wrong, it hasn't updated yet." The chosen source's value stays in trace. This subtle color difference signals which side was the outlier without reactivating the conflict.

6. Hold is a color shift. Same as v1: holding shifts the visual treatment from spectrum color to filed grey. All information preserved, urgency removed. Resume reverses it.

7. Backend status gap: "hold". Same as v1: the backend needs a "hold" status added to conflict and change items. POST /items/{id}/hold and POST /items/{id}/resume-review endpoints needed.

8. Buttons use the spectrum. Resolve → stamp. Hold → filed. Acknowledge → pencil. Resume → stamp. Every action carries the visual weight of what it produces.

9. Start Review is neutral. The "Start Review" button on Surface 2 transitions detected → in_review without resolving. It uses neutral styling (ink/rule-emphasis, no spectrum color) because it's a preliminary signal, not a resolution. It disappears once the item is already in_review. Visual treatment doesn't change between detected and in_review — intensity stays full. The status change is visible in the exec summary dock, signaling to other team members that someone is actively examining the item.
